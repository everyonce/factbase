# Audit: Maintenance Path Gaps

**Date:** 2026-03-15  
**Scope:** Large KB maintenance scenarios — scan efficiency, resolve loops, Obsidian sync, DB/file divergence, session limits

---

## Findings

| Scenario | Gap | Impact | Suggested Fix |
|---|---|---|---|
| **1. Scan of 1,400+ docs** | Files are processed in chunks of 100 (`SCAN_CHUNK_SIZE = 100`), with parallel I/O (`pre_read_files` uses `rayon::par_iter`) and per-chunk embedding. Deadline checking fires at chunk boundaries, not per-file. A 1,400-doc repo processes in 14 chunks; if a chunk takes longer than the remaining deadline budget, the scan is interrupted mid-chunk with `file_offset` set to resume. This is correct but the deadline is only checked *before* each chunk, so a slow embedding call inside a chunk can overshoot the budget by up to one chunk's worth of time. | Low — overshoot is bounded to one chunk (~100 docs × embedding latency). For Bedrock at ~200ms/doc, worst-case overshoot is ~20s. | Check deadline after each embedding batch inside `run_embedding_phase`, not just at chunk boundaries. |
| **2. Resolve of 5,000+ questions** | `resolve_step2_batch` loads ALL documents from disk on every call (`load_review_docs_from_disk`), parses every review section, and builds the full unanswered list before slicing a batch. For 5,000 questions across 1,400 docs, this is O(N) per batch call. With a default batch size of 30, that's ~167 calls × full doc scan each time. | Medium — each step-2 call re-reads and re-parses all files. On a 1,400-doc KB with disk I/O, this could add seconds per batch call. Total resolve time grows quadratically with question count. | Cache the unanswered question list in the DB (already partially done via `review_questions` table). Use the DB table as the source for batching instead of re-scanning disk on every call. |
| **3. File edited in Obsidian between scans** | The file watcher (`watcher.rs`) debounces at 500ms and triggers a full rescan. During the rescan, `full_scan` reads the disk file, computes a new hash, and detects it as `is_modified`. The review queue merge logic (`merge_review_queue`) then merges the DB's review section into the new disk content before writing to DB. This is correct. | None — handled correctly. The 500ms debounce means rapid edits are batched. | No fix needed. |
| **4. DB sync after Obsidian renames a file** | When Obsidian renames a file, it updates its own wikilinks but factbase's DB still holds the old `file_path`. The document ID (in frontmatter) is stable, so on the next scan the file is detected as `is_moved` (same ID, different path). The DB is updated with the new path. However, if the watcher triggers a rescan immediately after the rename, the old path is briefly "missing" (marked deleted) before the new path is found — this depends on whether the watcher fires once (rename event) or twice (delete + create). On macOS with `notify`, renames typically fire as a single `Rename` event, but the watcher calls `full_scan` which re-discovers all files, so the document is correctly detected as moved. | Low — there is a brief window where the document appears deleted if the watcher fires before the rename completes. The next scan corrects it. | No code change needed, but the Obsidian tip in `maintain_step7` ("If you renamed files in Obsidian since the last scan, run factbase(op=scan)") is the right mitigation. |
| **5. Review questions in files but not in DB (known bug)** | The auto-migration heuristic fires when `synced_docs < 80%` of docs with review sections. This means up to 20% of docs can have questions in files but not in DB without triggering a full re-sync. Additionally, the heuristic uses `known` (the pre-scan DB snapshot) to count docs with review sections — newly added docs in this scan are not yet in `known`, so their questions are synced via the per-doc path in `run_embedding_phase`, but only if `skip_embeddings=false`. In `skip_embeddings=true` mode, the sync is explicit. The 80% threshold is arbitrary and could miss edge cases where exactly 20% of docs are out of sync. | Medium — up to 20% of docs can have stale DB question counts without triggering re-sync. The `review_queue` MCP tool and `resolve_step2_batch` both prefer disk content (`load_review_docs_from_disk`), which mitigates this for the resolve workflow, but `get_review_queue` (the MCP op) queries the DB table directly and will miss disk-only questions. | Lower the threshold from 80% to 50%, or add a `reindex_reviews` flag to the maintain workflow's scan step. Also: `get_review_queue` should fall back to disk scanning when the DB count is zero for a repo that has files with review sections. |
| **5b. Other DB/file sync gaps** | (a) `has_review_queue` column in the `documents` table can be stale — it's set during upsert but not updated when questions are pruned by `strip_answered_questions` during scan. A doc that had all questions answered will still show `has_review_queue=TRUE` until the next full content update. (b) The `review_questions` table is only synced for *changed* documents during a normal scan (via `run_embedding_phase`). Unchanged documents are skipped entirely — their DB question records are never refreshed from disk. If a user manually edits a review section in a file without changing other content (e.g., adds a `> defer:` answer), the hash doesn't change, the doc is skipped, and the DB question record stays stale. | Medium — (b) is the more impactful gap: a user answering questions directly in Obsidian without triggering a content change will have their answers invisible to the DB until `reindex_reviews=true` is passed. | For (b): compute a review-section-specific hash and compare it separately from the full content hash. If the review section changed but the body didn't, sync questions without re-embedding. |
| **6. Session hits context limits mid-resolve** | The resolve workflow is designed for this: `continue=true` signals the agent to keep calling step 2; subsequent batches use a slim instruction (< 500 chars) to minimize context usage; the DB tracks answered questions so they never reappear; `load_review_docs_from_disk` re-reads disk on each call so partial answers written to disk are picked up. The `completion_gate` field gives the agent a compact progress indicator. The workflow explicitly tells agents "Your runtime compacts automatically" and "the DB tracks your progress — nothing is lost." | None for correctness. However, there is no server-side session state — if the agent loses its context entirely and starts a new session, it must call `workflow(resolve, step=1)` to re-orient. The `type_distribution` and `recommended_order` fields in step 1 give it everything it needs to resume. | No code change needed. Consider adding a `resume_hint` field to step 1 that summarizes progress (e.g., "X questions answered, Y remaining") to help agents that restart mid-session. |
| **7. Maintain with zero review questions** | `maintain_step6` explicitly checks `total_unanswered == 0` and returns early with `"No review questions found — the knowledge base is clean. Skip to the final report."` and `when_done: "Call workflow with workflow='maintain', step=7"`. This is correct. | None — handled correctly. | No fix needed. |

---

## Summary

The most impactful gaps are:

1. **Resolve loop O(N) per batch** — `load_review_docs_from_disk` re-scans all files on every step-2 call. For large KBs this adds latency that compounds over hundreds of batches.

2. **Review question DB/file divergence** — The 80% re-sync threshold and the skip of unchanged docs mean the `review_questions` DB table can be stale for up to 20% of docs. The resolve workflow mitigates this by reading disk directly, but `get_review_queue` (DB-only) does not.

3. **Review section hash not tracked separately** — Answering questions in Obsidian without changing body content leaves the DB stale until an explicit `reindex_reviews` scan.
