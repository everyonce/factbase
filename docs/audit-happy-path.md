# Happy Path Audit: New KB Creation to Daily Use

Audit date: 2026-03-15. Scope: new user journey from `create` workflow through daily `maintain`/`refresh` use.

## Methodology

Traced the workflow instruction chain in `src/mcp/tools/workflow/instructions.rs` and `mod.rs`, error handling in `src/mcp/tools/`, `src/scanner/`, and `src/error.rs`.

---

## Findings

| Step | Gap | User impact | Severity |
|------|-----|-------------|----------|
| **create step 1 (no domain)** | `DEFAULT_SETUP_INIT_INSTRUCTION` tells the agent to write `perspective.yaml` in step 1, but step 2 is also labelled "Configure Perspective" and does the same thing. The two steps overlap — an agent following step 1 literally will write the file, then step 2 overwrites it with a different template. | Agent may produce a perspective.yaml that doesn't match what step 3 validates, causing a confusing re-do loop. | Medium |
| **create step 3 (perspective validation)** | Validation only checks that the YAML parses and has known fields. It does not validate that `allowed_types` is non-empty, that `focus` is present, or that `stale_days` is a positive integer. An agent can write `allowed_types: []` and pass validation. | KB created with no document types; subsequent `check` generates no questions; user never notices until they try to organize. | Medium |
| **create step 3 (perspective validation)** | Error message on failure is: `"perspective.yaml is missing, empty, or has invalid YAML. Go back to step 2 and fix it."` — no indication of *what* is wrong (e.g., which field failed to parse, what line the YAML error is on). | Agent must guess what to fix; may loop on step 3 multiple times. | Low |
| **create step 4 (first documents)** | Instruction says "Create 2-3 example documents using `factbase(op='create')`" but `factbase(op='create')` does not exist — the correct op is `factbase(op='create')` which maps to `create_document`. The instruction is correct, but the `op` name in the tool schema is `create`, not `create_document`. This is fine. However, the instruction does not tell the agent to call `factbase(op='scan')` after creating documents — it defers that to step 5. If the agent calls `factbase(op='check')` before scanning, it gets 0 results with no explanation. | Agent may call check prematurely and see an empty result, not understanding why. | Low |
| **create step 5 (first scan of fresh KB)** | On a fresh KB with no prior embeddings, `scan_repository` returns `requires_confirmation: true` with `confirmation_reason: "full embedding rebuild"` and `confirmation_details: "... first-time generation"`. The `create` workflow step 5 instruction does not mention this confirmation gate — it just says "Call `factbase(op='scan')`". An agent that doesn't read the response carefully may not pass `force_reindex=true` or may stall waiting for user confirmation that never comes. | First scan silently stalls or agent loops. | High |
| **create step 5 (first scan of fresh KB)** | The scan instruction says "It WILL return `continue: true` with a `resume` token for any non-trivial repository." For a brand-new KB with 2-3 documents, the scan completes in one call and does NOT return `continue: true`. The word "WILL" is misleading for small KBs. | Agent may wait for a resume token that never arrives, or call scan again unnecessarily. | Low |
| **Bedrock/Ollama unavailable** | When the embedding provider is unreachable, `scan_repository` returns `FactbaseError::Embedding(...)`. The error message is the raw HTTP error from the provider (e.g., `"connection refused"` or `"AccessDeniedException"`). There is no actionable hint like "Run `factbase doctor` to check your embedding provider." | New user sees a cryptic error with no recovery path. | High |
| **Bedrock/Ollama unavailable** | The `create` workflow step 5 instruction does not mention what to do if the embedding provider is unavailable. There is no fallback suggestion (e.g., "If scan fails with an embedding error, run `factbase doctor` to diagnose"). | User is stuck with no guidance. | High |
| **Bedrock/Ollama unavailable** | `factbase doctor` is a CLI command only — it is not exposed as an MCP tool. An agent using factbase via MCP has no way to diagnose embedding provider connectivity. | Agent cannot self-diagnose; must ask user to run CLI command. | Medium |
| **add workflow (ingest, step 2 — research)** | The research step says "Use available research tools for recent, authoritative information." If the agent has no web search tool, it has no way to research and no guidance on what to do. The instruction does not say "If you have no research tools, skip to step 3 and create documents from your training knowledge, noting that sources will be weak." | Agent either halts or creates documents with no sources, generating a flood of `missing` review questions. | Medium |
| **add workflow (ingest, step 3 — create)** | The instruction warns "Never use 'Author knowledge' as a source" but does not explain what `author-knowledge/` files are or how to create them. A new user who wants to add facts from their own expertise has no path. | User cannot add their own knowledge without violating the source requirement. | Low |
| **maintain step 1 (scan with full rebuild)** | Same as create step 5: `requires_confirmation` gate is not mentioned in the maintain workflow scan instruction. The instruction says "Call `factbase(op='scan')` with `time_budget_secs=120`" with no mention of the confirmation gate. | Agent stalls on first maintain run after an embedding model change. | High |
| **maintain step 3 (check)** | The check instruction says "Call `factbase(op='check')` (one call — no paging needed)." This is correct, but the instruction does not explain what the `questions_total` output means for a brand-new KB. A new KB with 2-3 documents and no temporal tags will generate many `temporal` questions immediately. A new user may be alarmed by 20+ questions on a 3-document KB. | New user thinks something is broken; may abandon the KB. | Medium |
| **maintain step 3 (check)** | The temporal question filtering guidance says to dismiss low-confidence temporal questions about "stable feature descriptions or capability lists." This is domain-specific guidance that may not apply to all KBs (e.g., a historical events KB where all facts are time-sensitive). | Agent over-dismisses temporal questions in non-software domains. | Low |
| **resolve step 2 (answer loop)** | The instruction says "Every answer MUST cite an external source." For a brand-new KB where the user just added facts from their own knowledge, there are no external sources. The agent will defer everything, leaving the review queue full. There is no guidance for the case where the KB owner IS the authoritative source. | New user's KB is permanently stuck in "needs review" state. | High |
| **resolve step 2 (answer loop)** | The `believed` confidence level is described as "stays in the queue for human review and are NOT applied." But the resolve workflow step 3 says "Apply your answers by rewriting the documents directly." If all answers are `believed`, step 3 does nothing and the queue never clears. There is no guidance on what to do when all answers are `believed`. | User is stuck in an infinite resolve loop. | Medium |
| **refresh step 3 (research)** | The refresh instruction says "For each entity that needs refreshing (prioritize stale facts, then low temporal/source coverage)." For a brand-new KB, ALL entities have low coverage. The instruction does not cap the number of entities to research, so an agent may try to research all 2-3 documents exhaustively before moving on. | First refresh takes much longer than expected; agent may time out. | Low |
| **perspective.yaml missing required fields** | If `perspective.yaml` exists but is missing `focus` and `allowed_types`, the create workflow step 3 reports `perspective_status: "ok"` with `perspective_parsed: ""` (empty string). The agent sees "ok" and proceeds, but the KB has no focus or type constraints. | KB is created with no structure; all documents are untyped; organize and check produce poor results. | Medium |
| **perspective.yaml missing required fields** | The `perspective.yaml` format is documented in the workflow instruction but not validated against a schema. Fields like `stale_days: "never"` (string instead of integer) or `citation_patterns: "foo"` (string instead of array) are silently ignored rather than rejected. | User writes invalid YAML values; features silently don't work; no error message. | Medium |
| **No repository registered** | If the user calls `factbase(op='scan')` or `factbase(op='create')` without first registering a repository (via `factbase repo add` CLI or `factbase(op='init_repository')`), the error is `"No repository found. Register one with 'factbase repo add <id> <path>'"`. This is a CLI instruction inside an MCP context — the agent cannot run CLI commands. | Agent receives a CLI instruction it cannot follow; no MCP equivalent is suggested. | High |
| **No repository registered** | The `create` workflow step 1 says "The repository will auto-initialize when you run `factbase(op='scan')` in a later step." This is misleading — `factbase(op='scan')` does NOT auto-register a repository. The repository must be registered first via `factbase(op='init_repository')` (which is an MCP tool) or the CLI. The workflow does not call `init_repository` at any step. | Agent reaches step 5 and scan fails with "No repository found." | Critical |
| **init_repository MCP tool** | `init_repository` is listed in the MCP tools table in the README but is not in the `op_to_tool_name` dispatch map in `src/mcp/tools/mod.rs`. Calling `factbase(op='init_repository')` returns `"Unknown factbase op: init_repository"`. | Agent cannot initialize a repository via MCP; must use CLI. | Critical |
| **Daily use: scan after file edits** | After a user edits a file in Obsidian or another editor, the scan correctly picks up changes. However, if the user renames a file in Obsidian (which updates wikilinks), the factbase database still holds the old path. The scan will re-index the file at the new path and create a new document ID, leaving the old document as a ghost. The `maintain` workflow tip mentions this for Obsidian format only; non-Obsidian users get no warning. | Duplicate documents accumulate; links break; user doesn't know why. | Medium |
| **Daily use: duplicate title detection** | There is no validation that prevents two documents from having the same title. The link detection system uses title matching, so duplicate titles cause ambiguous links. No warning is shown when creating a document with a title that already exists. | Link graph becomes unreliable; `@q[ambiguous]` questions flood the queue. | Medium |
| **Daily use: invalid @t[] tag format** | The `create_document` and `update_document` tools do not validate `@t[...]` tag syntax. An agent can write `@t[Wolfgang Amadeus Mozart]` and it will be stored without error. The `check` command will not flag this as a corruption unless the `corruption` question type is triggered. | Invalid temporal tags silently corrupt the KB; temporal search returns wrong results. | Medium |
| **Error messages: generic Config errors** | When an unknown `op` is passed to `factbase`, the error is `"Unknown factbase op: <op>"`. When an unknown tool is called, the error is `"Unknown tool: <tool>"`. These are clear. However, when a required parameter is missing (e.g., calling `factbase(op='create')` without `path`), the error is `"Parse error: path is required"` — the `Parse error:` prefix is confusing in this context. | Agent may not understand that "Parse error" means "missing parameter." | Low |
| **Error messages: embedding errors** | `FactbaseError::Embedding` messages are raw provider errors. For Bedrock, this includes AWS SDK error codes like `"AccessDeniedException: User: arn:aws:iam::... is not authorized to perform: bedrock:InvokeModel"`. The error does not include the actionable hint from the README ("Enable model access in the Bedrock console"). | New user sees an IAM error with no recovery path. | High |

---

## Summary by Severity

**Critical (2):** Repository auto-initialization is not wired up — the `create` workflow claims scan will auto-register the repo, but it won't; and `init_repository` is not callable via MCP.

**High (6):** First-scan confirmation gate not mentioned in workflow instructions; embedding provider errors have no actionable hints; resolve loop has no path for owner-authored facts; "No repository found" error gives CLI instructions in MCP context; Bedrock IAM errors have no recovery guidance.

**Medium (9):** Perspective validation too permissive; duplicate titles not detected; `@t[]` syntax not validated; `believed` answers leave queue permanently full; Obsidian rename ghost documents; research step has no fallback for agents without web tools; maintain scan confirmation gate not mentioned; new KB generates alarming question count with no explanation.

**Low (7):** Misleading "WILL return continue:true" for small KBs; step 1/2 overlap in create workflow; perspective parse error gives no line number; `author-knowledge/` not explained; temporal question filtering is domain-specific; `believed` loop has no exit guidance; `Parse error:` prefix on missing-parameter errors.
